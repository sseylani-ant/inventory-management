Factory Inventory Management System

A full-stack demo application: a Vue 3 single-page app talking directly to a FastAPI backend that serves in-memory mock data loaded from JSON files. There is no database — everything is read into Python lists at startup.

Vue 3 + Vite FastAPI + Pydantic v2 In-memory JSON data pytest test suite

Data Flow

How a filter change on any view becomes rendered data — the same path for all pages.

  1. User picks a filter in FilterBar.vue (Time Period, Warehouse, Category, or Order Status). The value lands in useFilters.js — module-level refs shared as a singleton, so every view sees the same filter state instantly.
  2. The active view watches the filter state and calls the matching method on client/src/api.js, which serializes non-all filters into query params (e.g. /api/orders?warehouse=WH-A&month=Q3-2025).
  3. FastAPI handles the request: apply_filters() narrows by warehouse/category/status, filter_by_month() resolves months and quarters against order_date, then summary endpoints aggregate on the fly.
  4. Pydantic response models validate the outgoing shape; the JSON response returns to the browser.
  5. The view stores raw results in refs (allOrders, inventoryItems, …) and derives everything displayed — totals, groupings, chart series — through computed properties, so re-filtering re-renders without extra requests.

Observations Worth Knowing

Client methods without backend endpoints. api.js defines task CRUD (/api/tasks) and purchase-order creation/lookup (/api/purchase-orders), and main.py even declares a CreatePurchaseOrderRequest model — but no such routes exist on the server. TasksModal and PO creation from the Backlog flow will 404 until those endpoints are implemented.
Orphaned view. views/Backlog.vue is a complete backlog-management page, but it has no route in main.js and no nav link — it is currently unreachable.
Demo-grade settings. CORS allows all origins, the API binds 0.0.0.0, and auth is a mock composable. Fine for a demo; none of it is production-ready.
